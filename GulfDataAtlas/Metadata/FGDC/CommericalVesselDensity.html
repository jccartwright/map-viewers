Description:
Abstract:
These data were generated to provide insight into traffic patterns on a macro scale so that they could be analyzed across the coastal waters of the Continental United States. For this dataset, a transit is counted for each unique vessel intersecting a 1 kilometer square grid cell each day. This data represents the total number of vessel transits from October 2009 to October 2010. There were some grid cells which could not be processed, but we do not believe this interferes with the integrity of this dataset. Please note multiple connection errors occurred during the time frame of this study. Where possible, data gaps were filled by requests to the Coast Guard or other organizations that received the same data feed. However, due to resource constraints, uninterrupted coverage was not obtained. Overall data outages were minimal, on the order of less than a day per month. Because the outages were random and affected all areas uniformly, they do not have a significant effect on the integrity of the data. Also, as stated on the United States Coast Guard (USCG) Nationwide Automatic Identification System (NAIS) website: AIS data is not representative of all vessel traffic, and USCG NAIS receivers do not fully cover the entire extent of this study area. Please take time to understand both of these limitations. Refer to the NAIS website for more details: http://www.navcen.uscg.gov/?pageName=NAISmain.

To support coastal and ocean planning and other activities pursuant to the Energy Policy Act, Coastal Zone Management Act, Magnuson-Stevens Fishery Conservation and Management Act, National Environmental Policy Act, Rivers and Harbors Act and the Submerged Lands Act.

Process_Step:
Process_Description:
Source data derived from the raw AIS data processing is as follows: The USCG maintains a network of AIS receivers that collects AIS messages from passing ships. These messages are transmitted to the USCG data center that compiles the data and provides data feeds to other government agencies. In accordance with the USCG COMDTINST 5230.80, the USCG provided OCS with a “Level A” data feed. Level A is unfiltered real-time data that is less than 96 hours from initial time of transmission. OCS has subscribed to this data since 2008. To limit data storage requirements, the data feed was filtered by the USCG; sending one position message per ship per minute. All duplicate messages (i.e. ship broadcasts received by more than one NAIS station) were removed. This real time feed was archived at OCS. A specialized software NOAADATA.py (see K. Schwehr. The noaadata-py Software Tool-set, v0.42, 2009. http://vislab-ccom.unh.edu/~schwehr/software/noaadata/) was used to create daily files and to load the data into an Oracle Spatial database. Despite receiving this filtered feed, a great deal of conditioning needed to take place to prepare this data to be analyzed. The AIS system was not intended or designed for subsequent analysis; however Calder and Schwehr proved that given the proper conditioning, AIS data can be used for traffic analysis. (see B. R. Calder., K. Schwehr. Traffic Analysis for the Calibration of Risk Assessment Methods. Proceedings: US Hydrographic Conference 2009, Norfolk, VA, 11-14 May 2009, http://www.thsoa.org/us09papers.htm) Based on their research, we filtered out AIS messages with non-unique user IDs and vessels with erroneous dimensions. We also separated messages by speed, separating those that reported a speed of less than 0.4 knots into a separate anchored table. The accuracy and abundance of the AIS data support high resolution analysis specifically within port areas. Our study, however, was interested in traffic patterns on a macro scale to allow patterns to be analyzed across the coastal waters of the Continental United States. To limit the processing time and the overall file size for each region, a grid cell size of 1 kilometer was chosen. For this dataset a "transit" is counted for unique vessels intersecting a grid cell each day. Multiple trips into a grid cell on a given day by the same vessel were counted as one transit. Instead of calculating transits based on the coordinates within the AIS message, transit lines were created by connecting all the vessels reports each day. Traffic counts were then calculated by summing the number of lines within each cell. Although this dataset only contains the total traffic count “TRNSTS_TTL” the original dataset has transits by AIS vessel type and many other attributes as well. These were processed from May - July 2012.

Process_Step:
Process_Description:
Futher data development processing steps were followed to obtain this feature class: Acquired original source data and maintained a copy on the D-tier (data01_d), \\csc-s-san1b\marinsp\MultipurposeMarineCadastre\original\2012\OCS AIS GRIDS. Created a geodatabase in a personal workspace and imported the original shapefiles into this file geodatabase format. Added a field in each of the feature classes on which to base classification ranges and deleted all values of "0" in each of the data sets. Using the field calculator, populated the new classification field values based on selecting the values of 'TRNSTS_TTL' field; this was performed on each UTM zone feature class. Then performed a "dissolve" on each of the individual UTM zones datasets based on the classification field. Created a new feature class with NAD 83 projection, added a field for the classification code, and loaded the data from the classified dissolved dataset into the new NAD 83 dataset. Validated the feature class geometry using "Data Management > Features > Repair Geometry". Added a field in each of the three new data sets called 'vesselCountRange’, populated this new field using field calculator to display the actual range of values within that classification code. When necessary, opened ArcMap and in a new MXD, clicked File > Map Document Properties and checked the box “Store relative path names to data sources”. Added in the Oceans Basemap in the working MXD to work up the color scheme for the data. Selected the fields to make visible (chose to hide Classification Code and show Vessel Count Range). Symbolized the layer by Classification Code field to represent the Vessel Count Ranges. Developed a layer file containing symbology from each feature class. Included Layer Name, Description, Credits, and Scale Ranges (if applicable) within the layer file and its source. Arranged quality process review by peers. Reviewed quality process feedback, determined applicability, and made adjustments to data to meet process specifications. Placed a copy of the feature class/es within ArcSDE. Projected the symbolized layer into Web Mercator and placed within the viewer's MXD. Worked with the MXD steward to determine where and how the layer fit within the MXD. Acquired, created and updated metadata from providers (as needed). Planned to annually evaluate source data to determine requirement for replacement.
